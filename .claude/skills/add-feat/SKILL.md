# Skill: add-feat — 从 Prompt 生成 Feature 开发计划并落盘到 `vibe/`

## 1. 用途（什么时候用这个 skill）

当用户通过一段自然语言 prompt 描述一个想做的新功能 / 新特性 / 新模块（"我想加一个 X"、"帮我设计一下 Y"、"给我一个 Z 的开发计划"等）时，使用本 skill：

- 把模糊的想法转成一份**可执行、可评审**的开发计划
- 计划遵循固定的 7 节结构（见 §4 模板）
- 计划以 Markdown 文件形式写入 **`vibe/` 目录**，文件名体现功能主题，方便后续迭代和评审

本 skill 不负责实际写代码，只负责产出**计划文档**。真正的实现可由后续的编码 agent（或人工）按计划执行。

## 2. 核心原则

1. **先读懂 prompt，再动笔**：如果 prompt 关键信息缺失（例如：目标模块不清、现有实现位置不明），先用 `Grep` / `Glob` / `SemanticSearch` / `Read` 等工具在仓库中定位上下文，再开始写计划。
2. **7 节结构不可缺项**：即使某些节信息暂缺，也必须保留该节并显式标注 `TODO / 待补充 / N/A`，而不是直接删节。
3. **文件路径要真实**：计划里引用的 `path/to/file.py` 必须是仓库中真实存在或将要创建的路径，不要写占位路径。可以用工具先验证。
4. **写入而非覆盖**：写入 `vibe/` 前，先检查目标文件是否已存在；若已存在，询问用户是覆盖、追加版本号（如 `xxx.v2.md`），还是合并。
5. **完成后告知路径**：写完后必须把最终落盘路径明确告诉用户。

## 3. 工作流（必须按顺序执行）

### Step 1: 解析 prompt
- 从用户 prompt 中抽取：功能名称、目标模块、痛点、预期效果、限制条件
- 列出不确定点（后面要么查代码，要么在计划里标 TODO）

### Step 2: 调研仓库上下文（按需）
- 用 `Glob` 定位相关文件 / 目录
- 用 `Grep` / `SemanticSearch` 搜关键接口、现有实现
- 用 `Read` 精读关键文件
- 目标：把"需要修改 / 新增 / 禁止修改的文件"这一节写实

### Step 3: 确认 `vibe/` 目录存在
- 用 `Shell` 执行 `ls vibe/ 2>/dev/null || mkdir -p vibe` 创建（如不存在）
- 如果目录里已有同主题文件，按 §2 原则 4 处理

### Step 4: 决定文件名
- 命名格式：`vibe/<kebab-case-feature-name>.md`
- 例子：`vibe/add-pipeline-parallel-for-wan21.md`、`vibe/refactor-sampler-api.md`
- 名称要能一眼看出做什么，避免 `vibe/feat1.md` 这种无信息名

### Step 5: 按 §4 模板填写计划
- 标题改为 `# Feature: <具体功能名>`（去掉方括号占位符）
- 每一节都要有实质内容；实在没有就显式写 `N/A` 或 `TODO: <需要澄清的问题>`
- 对于"实施步骤"，按合理的工程顺序（接口 → 实现 → 测试 → 文档）拆成 3~8 个 Step
- 对于"验收标准"，至少覆盖：功能测试、回归测试、代码质量三类

### Step 6: 用 `Write` 工具写入 `vibe/<file>.md`
- **不要用 `echo` / `cat > file` 等 shell 写文件**，使用 `Write` 工具
- 写入后用 `Read` 或 `ls` 验证文件已生成

### Step 7: 向用户汇报
- 明确给出文件路径（如 `vibe/add-pipeline-parallel-for-wan21.md`）
- 简述计划里的 3~5 个关键决策点（新旧方案差异、关键风险等）
- 询问是否需要调整某一节

## 4. 计划模板（写入 `vibe/<xxx>.md` 时使用）

````markdown
# Feature: [功能名称]

## 1. 背景（是什么 && 为什么）

### 现状分析
- 当前系统的状态
- 存在的问题或痛点
- 为什么需要这个功能

### 设计问题
- 具体的技术问题
- 架构层面的不足
- 用户体验的缺陷

## 2. 目标（想做成什么样子）

### 核心目标
- 主要目标 1
- 主要目标 2
- 主要目标 3

### 预期效果
- 功能层面的改进
- 性能层面的提升
- 用户体验的优化

## 3. 范围明确（需要修改与新增那些文件，不能修改和新增那些文件）

### 需要修改的文件
- `path/to/file1.py` - 修改原因和改动范围
- `path/to/file2.py` - 修改原因和改动范围

### 需要新增的文件
- `path/to/new_file1.py` - 新增原因和功能说明
- `path/to/new_file2.py` - 新增原因和功能说明

### 不能修改的文件
- `path/to/stable_api.py` - 原因：公共 API，需保持兼容
- `path/to/legacy_code.py` - 原因：遗留代码，风险高

### 影响范围
- 依赖此功能的模块
- 可能受影响的下游系统

## 4. 设计方案（大致的方案）

### 新旧方案对比
- **旧方案**：简要描述现有实现
- **新方案**：简要描述新设计
- **核心变化**：突出关键改进点

### 架构/接口设计
- 类型定义
- API 设计
- 核心类/函数签名

### 核心操作流程
- 关键流程说明
- 时序图/流程图（如需要）
- 数据流转说明

### 技术细节
- 类型注解
- IDE 支持
- 其他重要实现细节

## 5. 实施步骤

### Step 1: [步骤名称]
- 具体操作 1
- 具体操作 2
- 涉及文件清单

### Step 2: [步骤名称]
- 具体操作 1
- 具体操作 2
- 涉及文件清单

### Step N: [步骤名称]
- 具体操作 1
- 具体操作 2
- 涉及文件清单

### 兼容性与迁移
- 向后兼容策略
- 迁移路径
- 过渡期安排

## 6. 验收标准

### 功能测试
- 测试用例 1：具体场景和预期结果
- 测试用例 2：具体场景和预期结果
- 测试用例 N：具体场景和预期结果

### 代码质量
- 类型检查通过
- 代码覆盖率达标
- 代码规范符合要求
- 注释和文档完善

### 回归测试
- 现有功能正常运行
- API 兼容性验证
- 性能无明显退化

### 性能与可维护性
- 性能指标（如适用）
- 可维护性评估
- 技术债务清理

### 文档与示例
- 用户文档更新
- 开发者文档更新
- Demo 和示例代码

## 7. 其他说明

### 相关资源
- Issue 链接
- 参考设计文档
- 相关讨论记录

### 风险与应对
- 潜在风险 1 及应对措施
- 潜在风险 2 及应对措施

### 依赖关系
- 依赖的其他 Feature
- 被依赖的场景
````

## 5. 质量自检（写完计划后，至少过一遍）

- [ ] 标题已替换为具体功能名，不是 `[功能名称]` 占位符
- [ ] 7 节全部存在，没有整节被删
- [ ] "需要修改 / 新增 / 不能修改"里的路径都是仓库真实路径或明确新建路径
- [ ] "新旧方案对比"里明确写出了核心差异，而不是空话
- [ ] 实施步骤至少 3 个，且顺序合理（通常是"接口 → 实现 → 测试 → 文档"）
- [ ] 验收标准至少覆盖：功能测试、回归测试、代码质量
- [ ] 风险一节有实质内容，不是"暂无"
- [ ] 文件已落盘到 `vibe/<name>.md`，路径已汇报给用户

## 6. 常见反模式（不要这样做）

- ❌ 直接把用户的 prompt 原样丢进"背景"，不做任何调研和结构化
- ❌ 用 `path/to/xxx.py` 这种假路径交差
- ❌ "实施步骤"只写一句"按计划实现"
- ❌ 把计划写到 `docs/`、`docs_always/`、根目录等其他位置——必须是 `vibe/`
- ❌ 用 `echo "..." > vibe/xxx.md` 写文件（格式会被 shell 转义破坏，必须用 `Write` 工具）
- ❌ 写完不告诉用户文件路径
